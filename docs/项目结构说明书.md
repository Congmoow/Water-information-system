# 水利信息管理系统项目结构说明书

## 1. 文档说明

本文档用于说明水利信息管理系统的整体技术结构、系统架构、目录组织方式、模块边界和分层职责，作为项目开发、维护和答辩展示时的结构依据。项目背景、建设目标和业务范围详见《项目规格需求说明书》。

## 2. 项目概述

本项目采用前后端分离方式构建水利信息管理系统，本文档仅聚焦系统实现层面的技术结构、代码组织和部署组成，不展开项目背景、建设目标和功能需求说明。

## 3. 技术栈结构

### 3.1 后端技术栈

- ASP.NET Core Web API：提供 RESTful 接口
- Entity Framework Core：负责 ORM 与数据库访问
- SQL Server：负责关系型数据存储
- JWT：负责登录认证与接口鉴权
- ASP.NET Core OpenAPI：负责接口规范生成
- Scalar：负责交互式接口文档展示

### 3.2 前端技术栈

- Vue 3：前端核心框架
- TypeScript：增强类型安全与可维护性
- Vite：前端构建工具
- Vue Router：负责路由管理
- Pinia：负责状态管理
- Axios：负责接口请求封装
- Element Plus：负责后台管理界面组件
- ECharts：负责图表展示
- Leaflet：负责地图展示

### 3.3 开发工具

- Visual Studio 2022 / Rider / VS Code
- SQL Server Management Studio
- Postman
- Git

上述技术栈在系统中的职责划分为：前端负责展示与交互，后端负责接口与业务处理，数据库负责业务数据持久化存储。

## 4. 系统总体架构

### 4.1 架构模式

系统采用前后端分离架构。

前端负责页面渲染、用户交互、图表展示和地图展示；后端负责业务逻辑处理、权限认证、数据库访问与接口输出；数据库负责存储用户、工程、站点、监测和告警等核心业务数据。

### 4.2 架构流程

```text
Vue 3 + TypeScript 前端
        ↓ Axios 请求
ASP.NET Core Web API
        ↓ EF Core
     SQL Server 数据库
```

### 4.3 架构特点

- 前后端职责清晰，便于协作开发
- 分层结构明确，便于维护和扩展
- 支持模块化开发，适合课程项目分阶段实施
- 兼顾展示效果与实现复杂度，适合答辩演示

## 6. 后端项目结构设计

后端采用分层架构与模块化设计。

### 6.1 后端总结构

```text
backend/
├─ WaterInfoSystem.sln
└─ src/
   ├─ WaterInfoSystem.API/
   ├─ WaterInfoSystem.Application/
   ├─ WaterInfoSystem.Domain/
   ├─ WaterInfoSystem.Infrastructure/
   └─ WaterInfoSystem.Shared/
```

### 6.2 各层职责说明

#### API 层

负责请求接收、参数绑定、响应输出、中间件配置和接口文档配置，不承载核心业务逻辑。

主要内容包括：

- Controllers
- Middleware
- Filters
- Extensions
- Program.cs

#### Application 层

负责业务用例编排、服务接口定义、DTO 设计、参数校验、权限判断和事务协调。

主要内容包括：

- Contracts
- Interfaces
- Services
- Validators
- Mappings
- DependencyInjection

#### Domain 层

负责核心实体、枚举、领域状态和业务规则定义，不依赖数据库实现和 Web 框架。

主要内容包括：

- Entities
- Enums
- Constants

#### Infrastructure 层

负责数据库上下文、EF Core 配置、仓储实现、JWT 实现和其他基础设施能力。

主要内容包括：

- Persistence
- Repositories
- Identity
- Services
- DependencyInjection

#### Shared 层

负责系统通用结果封装、常量、异常类型和工具类，不放具体业务逻辑。

主要内容包括：

- Results
- Exceptions
- Constants
- Helpers

### 6.4 后端引用关系

```text
API → Application
API → Shared

Application → Domain
Application → Shared

Infrastructure → Application
Infrastructure → Domain
Infrastructure → Shared

Domain → （不依赖其他项目）
Shared → （不依赖业务项目）
```

### 6.5 后端结构约束

- Controller 只作为接口入口
- Service 不直接依赖 DbContext
- Repository 只负责数据访问
- DTO 与 Entity 分离
- Domain 不依赖 Infrastructure
- Shared 不承载业务服务

## 7. 前端项目结构设计

前端按业务模块与公共能力组合组织，避免页面、组件和接口文件混杂。

### 7.1 前端总结构

```text
frontend/
└─ water-info-web/
   ├─ public/
   ├─ src/
   ├─ .env.development
   ├─ .env.production
   ├─ package.json
   ├─ tsconfig.json
   ├─ vite.config.ts
   └─ index.html
```

### 7.2 前端详细结构

```text
src/
├─ api/
│  ├─ modules/
│  └─ http.ts
├─ assets/
├─ components/
│  ├─ common/
│  ├─ business/
│  └─ charts/
├─ composables/
├─ constants/
├─ enums/
├─ layout/
├─ router/
├─ stores/
├─ types/
├─ utils/
├─ views/
├─ App.vue
└─ main.ts
```

### 7.3 前端结构原则

- api/modules 按业务模块拆分接口文件
- views 下按业务模块拆分页与私有组件
- components/common 仅放真正全局复用组件
- composables 负责抽离可复用逻辑
- stores 按认证、用户、权限、应用状态等职责拆分
- types 统一维护接口和页面使用的数据类型

## 8. 模块化组织方式

以水库模块为例，推荐采用贯穿前后端的一致模块结构：

```text
Reservoir
├─ API：api/modules/reservoir.ts
├─ 页面：views/reservoir/list、form、detail
├─ 私有组件：views/reservoir/components/
├─ 业务服务：ReservoirService.cs
├─ 仓储接口：IReservoirRepository.cs
├─ 仓储实现：ReservoirRepository.cs
├─ 实体：Reservoir.cs
└─ 控制器：ReservoirController.cs
```

该方式能够保证模块边界清晰、职责稳定、便于功能扩展和代码定位。

## 9. 数据结构概要

系统核心数据结构围绕“用户—工程—站点—监测数据—告警记录”展开，主要表包括：

- Users
- Reservoirs
- Rivers
- Stations
- MonitoringDatas
- AlarmRecords

数据结构设计为后续业务模块、图表分析和地图展示提供统一的数据基础。

## 10. 部署结构说明

### 10.1 后端运行结构

- 使用 ASP.NET Core Web API 提供服务
- 通过配置文件维护数据库连接字符串
- 通过迁移管理数据库结构

### 10.2 前端运行结构

- 使用 Vite 启动前端开发环境
- 通过 Axios 与后端接口通信
- 通过路由和状态管理组织页面行为

### 10.3 数据库运行结构

- 使用 SQL Server 承载业务数据
- 通过初始化脚本、迁移和种子数据支持项目落地
