# 水利信息管理系统项目规格需求说明书

## 1. 文档说明

本文档用于明确水利信息管理系统的建设目标、使用范围、功能需求、非功能需求、数据需求、接口需求和实施要求，作为项目开发、验收和答辩说明时的需求依据。

## 2. 项目背景

随着信息技术的发展，传统水利管理方式逐渐暴露出数据分散、管理效率低、信息更新不及时、预警能力不足等问题。为提升水利工程管理水平，实现对水库、河道、监测站点、监测数据和告警信息的统一管理，需要建设一套基于 Web 的水利信息管理系统。

本项目面向课程项目场景，强调业务链路完整、结构清晰、便于开发和展示。

## 3. 建设目标

系统建设目标如下：

1. 实现水利工程基础信息的统一管理。
2. 实现监测站点信息的集中管理与展示。
3. 实现监测数据的录入、查询与统计分析。
4. 实现异常数据告警与告警记录管理。
5. 实现基于地图的工程与站点空间分布展示。
6. 构建一套结构清晰、易于扩展、适合课程项目展示与答辩的完整系统。

## 4. 项目定位

本项目定位为课程项目级水利信息管理系统，主要特征如下：

- 具备较完整的信息管理系统功能
- 采用前后端分离架构，满足系统实现要求
- 页面展示直观，适合演示和答辩
- 控制开发复杂度，保障可完成性

## 5. 适用对象与使用范围

### 5.1 适用对象

- 课程项目开发人员
- 项目演示与答辩人员
- 系统测试人员
- 指导教师或评审人员

### 5.2 使用范围

- 水利工程基础信息管理
- 监测站点信息管理
- 监测数据录入与查询
- 告警生成与处理
- 图表统计展示
- 地图空间展示

## 6. 功能需求

### 6.1 用户与权限管理需求

系统应支持以下能力：

- 用户登录
- 用户退出
- 获取当前用户信息
- 区分管理员与普通用户权限

### 6.2 水利工程信息管理需求

系统应支持以下能力：

- 水库信息新增、编辑、删除、查询
- 河道信息新增、编辑、删除、查询
- 预留泵站信息管理扩展能力
- 预留闸门信息管理扩展能力

### 6.3 监测站点管理需求

系统应支持以下能力：

- 监测站点新增、编辑、删除、查询
- 维护站点类型
- 维护站点状态
- 维护站点坐标信息

### 6.4 监测数据管理需求

系统应支持以下能力：

- 水位数据管理
- 雨量数据管理
- 流量数据管理
- 按时间范围查询历史监测数据
- 按站点查询监测数据

### 6.5 告警管理需求

系统应支持以下能力：

- 超阈值自动告警
- 告警记录列表查询
- 告警状态处理
- 告警等级展示

### 6.6 数据统计分析需求

系统应支持以下能力：

- 水位趋势图
- 雨量统计图
- 告警数量统计图
- 站点状态统计图

### 6.7 地图展示需求

系统应支持以下能力：

- 水利工程点位展示
- 监测站点点位展示
- 点位点击查看详细信息

## 7. 非功能需求

### 7.1 界面需求

- 系统界面应简洁、清晰、易操作
- 页面布局应适合课程演示场景
- 图表和地图应具备良好的可读性

### 7.2 性能需求

- 常规列表查询应在可接受时间内完成响应
- 页面切换与主要数据加载过程应保持流畅
- 系统应满足课程项目演示场景下的并发访问需求

### 7.3 可维护性需求

- 前后端职责应清晰分离
- 项目目录结构应规范
- 数据模型、接口和页面逻辑应具备良好的可定位性
- 模块之间应保持低耦合，便于后期扩展

### 7.4 可扩展性需求

- 系统应支持新增工程类型
- 系统应支持新增监测数据类型
- 系统应支持扩展更复杂的告警规则
- 系统应支持后续增加更细粒度权限控制

### 7.5 安全性需求

- 系统应支持基于 JWT 的身份认证
- 未认证用户不得访问受保护资源
- 不同角色用户应受到权限范围约束
- 密码信息应采用哈希方式存储

## 8. 数据需求

### 8.1 核心数据实体

系统至少应包含以下核心数据实体：

- 用户
- 水库
- 河道
- 监测站点
- 监测数据
- 告警记录

### 8.2 核心数据项要求

#### 用户

- 用户名
- 密码哈希
- 真实姓名
- 角色
- 创建时间

#### 水库

- 名称
- 所在位置
- 容量
- 管理单位
- 描述

#### 河道

- 名称
- 长度
- 流域
- 描述

#### 监测站点

- 站点名称
- 站点类型
- 经度
- 纬度
- 状态
- 描述

#### 监测数据

- 站点编号
- 数据类型
- 数据值
- 采集时间

#### 告警记录

- 站点编号
- 告警类型
- 告警等级
- 告警信息
- 告警状态
- 告警时间

## 9. 接口需求

### 9.1 认证接口

```http
POST /api/auth/login
GET /api/auth/profile
```

### 9.2 水库管理接口

```http
GET    /api/reservoir
GET    /api/reservoir/{id}
POST   /api/reservoir
PUT    /api/reservoir/{id}
DELETE /api/reservoir/{id}
```

### 9.3 河道管理接口

```http
GET    /api/river
GET    /api/river/{id}
POST   /api/river
PUT    /api/river/{id}
DELETE /api/river/{id}
```

### 9.4 站点管理接口

```http
GET    /api/station
GET    /api/station/{id}
POST   /api/station
PUT    /api/station/{id}
DELETE /api/station/{id}
```

### 9.5 监测数据接口

```http
GET    /api/monitoring
GET    /api/monitoring/history
POST   /api/monitoring
```

### 9.6 告警管理接口

```http
GET    /api/alarm
GET    /api/alarm/{id}
POST   /api/alarm
PUT    /api/alarm/{id}/handle
```

### 9.7 接口返回格式要求

系统接口建议统一采用如下结构：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

分页返回建议采用如下结构：

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "items": [],
    "total": 0
  }
}
```

## 10. 业务规则要求

### 10.1 权限规则

- 管理员具备全部或主要业务操作权限
- 普通用户主要具备查询和查看权限

### 10.2 告警规则

- 监测数据超过设定阈值时应生成告警记录
- 告警记录应具备等级和处理状态
- 已处理与未处理状态应可区分展示

### 10.3 数据查询规则

- 监测数据应支持按站点查询
- 监测数据应支持按时间范围查询
- 业务列表应支持基础查询和详情查看

## 11. 开发约束

### 11.1 后端约束

- Controller 不写业务逻辑
- Service 不直接依赖 DbContext
- Repository 只负责数据访问
- DTO 与 Entity 分离
- 各业务模块应独立组织

### 11.2 前端约束

- 页面只负责展示和交互
- 复杂逻辑应抽离到 composables 或 api 层
- 接口请求按业务模块拆分
- 全局组件与业务组件分离
- 类型定义应统一维护

## 12. 实施与测试要求

### 12.1 实施顺序要求

建议实施顺序如下：

1. 项目初始化
2. 登录与权限
3. 基础信息管理
4. 监测数据管理
5. 告警管理
6. 地图展示
7. 系统完善

### 12.2 测试要求

系统应覆盖以下测试内容：

- 登录功能测试
- 用户权限测试
- 水库、河道、站点 CRUD 测试
- 监测数据查询测试
- 告警处理测试
- 地图点位展示测试
- 接口返回和 Token 鉴权测试

## 13. 验收标准

系统达到以下条件时，可认为满足课程项目基本要求：

- 完成核心模块开发并可正常运行
- 登录认证流程完整
- 基础信息与监测数据可正常维护和查询
- 告警记录可生成、查询和处理
- 图表和地图页面可正常展示
- 页面结构清晰，便于演示和答辩说明

## 14. 后续扩展需求

系统后续可扩展以下能力：

- 泵站与闸门管理
- 数据导入导出
- 更复杂的权限控制
- 监测数据自动采集
- 更丰富的统计分析与告警规则
- 更完整的 GIS 展示能力
